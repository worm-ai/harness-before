**当前判断**

这个项目不是普通任务管理 CLI，而是一个“Git-native 的 AI 协作开发治理工具”。它现在已经完成了最小闭环：`plan / verify / audit / close / memory / route / drift`，README 也明确了这些能力 [README.md](/Users/cc/ai/harness-before/README.md:17)。核心实现集中在 `abh/core.py`，状态机、审计、关闭、路由和漂移分析都在这里 [core.py](/Users/cc/ai/harness-before/abh/core.py:41)。

项目最大的优势是方法论清晰、dogfooding 足、闭环已能跑通。最大的风险是“文档权威性”和“工具权威性”开始分裂：`task-board` 还停在 Sprint 5，但计划和代码已经到 Sprint 7，这个问题其实已经被 memory 明确记录 [mem-doc-sync-001.md](/Users/cc/ai/harness-before/docs/memory/mem-doc-sync-001.md:18)。

**长期定位**

建议把 ABH 的长期目标定为：

> 成为 AI 协作开发中的本地优先、证据ß优先、可审计的轨迹控制层。

它不应该变成完整项目管理平台，也不该太早做 Web UI。短中期最有价值的是把“计划、验证、审计、记忆、漂移”这套工作流变成任何仓库都能无痛接入的 CLI 和 Agent 协议。

**阶段 2：Agent Protocol 基础，1-2 个月**

当前 `abh` 已经具备可参数化 CLI，但还没有 Agent Protocol：没有 MCP Server、没有统一 JSON 输出模式、没有稳定的工具 schema，所有输出仍主要面向人类阅读。下一阶段应该先把 CLI 变成 AI Agent 可程序化调用的稳定接口，再在其上封装 MCP。

建议版本：`v0.2.0`

核心能力：

1. 为核心读操作增加机器可解析输出，例如 `--json` 或等价 JSON mode。
2. 定义稳定的 Agent tool schema：plan、verify、audit、memory、route、drift、doctor 的输入、输出和错误格式。
3. 统一 CLI 返回码和结构化错误，保证 Agent 能区分成功、校验失败、业务阻断和系统错误。
4. 增加只读 MCP Server 第一版，让 Claude、Cursor 等 Agent 能读取计划、审计、记忆、漂移和当前路线。
5. 在只读 MCP 稳定后开放受控 MCP 写工具，覆盖创建 plan、状态流转、记录 verify/audit、关闭计划、写 memory 和写 drift report。
6. MCP 写工具必须显式要求 `confirm=true`，并继续复用现有 ABH 门禁，不能绕过状态机、验证、审计或 doctor。

这个阶段的目标不是替代 CLI，而是把 CLI 变成 Agent Protocol 的底座。CLI 先稳定，MCP 才有可靠封装对象。

当前状态：已完成。`plan-012-agent-protocol-foundation`、`plan-013-json-output-and-errors`、`plan-014-readonly-mcp-server` 和 `plan-015-controlled-mcp-write-tools` 已关闭，ABH 已具备 Agent Protocol 基线、JSON CLI contract、结构化错误、只读 MCP Server 和受控 MCP 写工具，可作为 `v0.2.0` 里程碑。

阶段 3 的目标不是证明功能绝对正确，而是证明“在某个时间、某个 git 状态、某个本地环境下，ABH 确实执行过这些验证命令，并留下可复现、可审计的机器证据”。

建议版本：`v0.3.0`

核心能力：

2. 支持保存 stdout/stderr 摘要、退出码、耗时、artifact，以及 git commit、dirty status、cwd、ABH 版本、Python 版本、命令 argv、timeout 和环境变量 allowlist。

**阶段 4：Agent-First 吸引子入口层，2-4 个月**

目标：让 Codex、Claude Code、MCP 客户端和新仓库在 5 分钟内进入 active attractor -> plan -> verification -> audit -> memory 的轨迹控制回路，并让吸引子从文档约定升级为 CLI 可管理的一等对象。

这是当前最高优先级阶段。v0.3 已经证明 ABH 自身闭环可用；下一步不是做普通 onboarding，也不是孤立补一个 Attractor Registry，而是把“进入仓库”“进入 Agent”“进入计划”“进入关闭门禁”都绑定到 active attractor。ABH CLI 首先是 Agent 可调用的轨迹控制接口，其次才是人类手动操作工具；便利性必须服务 ABH 的核心吸引子：Git-native、证据优先、生成与验收分离的轨迹控制框架。

角色边界：

1. 人类主要负责定义或批准 active attractor、确认写入项目指令/hook/配置、执行或委托独立审计、批准 release 等高风险边界动作。
2. Agent 主要负责高频调用 ABH 命令：读取 active attractor、创建或更新 plan、运行 verification、生成 audit bundle/prompt、写入 memory、调用 `abh next --json` 决定下一步。
3. CLI 命令必须默认支持非交互、可重复、可脚本化和 JSON 输出；写入命令必须有 `--dry-run`、`--write`、`--confirm` 或等价确认边界，避免 Agent 无意越权。

建议版本：`v0.4`

核心能力：

1. `abh attractor list/show/create/supersede`：把 active attractor 从文档约定升级为 CLI 可管理对象，并记录版本差异、升级原因、影响范围和迁移策略。
2. active attractor 校验：plan 进入 `ready` 前必须引用 active attractor；route/drift/agent setup 优先读取 active attractor，而不是只靠固定路径和关键词。
3. `abh init`：在任意仓库生成 `.abh/`、`docs/plans/`、`docs/audits/`、`docs/memory/`、默认 active attractor、模板和最小 README 片段；初始化结果必须让用户直接进入第一条 plan / verify / audit / close 闭环。
4. `abh agent setup claude-code --json` 和 `abh agent setup codex --json`：输出对应 Agent 可直接消费的 setup bundle，包括项目指令、ABH 工作流提示词、命令执行约束、推荐 hook profile 和后续动作；这些指令必须写清“先读 active attractor、没有 plan 不开工、verification 不等于完成、close 依赖独立 audit、失败假设写 memory”。
5. `abh agent setup mcp --json`：输出通用 MCP 客户端可解析的 server 配置、工具清单和安全说明。`claude-code`、`codex`、`mcp --json` 是同一个 Agent Contract Export 能力的不同出口：前两者面向具体 Agent 产品，后者面向通用协议和机器可读配置；实际写入 AGENTS.md、CLAUDE.md 或配置片段时必须通过 `--write --confirm` 或等价确认。
6. `abh hooks install`：提供 `solo`、`team`、`strict` profile；hook 保护 ABH 不变量，例如 commit message 关联 plan、提交前 `abh doctor`、verification stale 提示、closing/closed audit gate，而不是做重型流程管控。
7. `abh next --json`：作为 Agent 的默认导航入口，根据当前仓库状态返回下一步 ABH 动作、推荐命令、是否需要人类确认和原因，例如补 active attractor、补 exit criteria、运行 verification、request audit、transition closing 或 close。
8. `abh onboarding check --json`：检查仓库是否 ABH-ready：active attractor、agent 指令、MCP/CLI 可用性、hook 配置、模板和至少一个完整闭环示例。
9. 新增 `docs/quickstart.md` 和 `docs/recipes/`，提供 5 分钟上手路径、Claude Code/Codex 使用配方、MCP 配置示例、Git hook profile 说明和第一个闭环 demo。
10. 优化分发路径：准备 PyPI / `uvx abh` 发布，保留 `uvx --from git+...` 路径。
