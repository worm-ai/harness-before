**当前判断**

这个项目不是普通任务管理 CLI，而是一个“Git-native 的 AI 协作开发治理工具”。它现在已经完成了最小闭环：`plan / verify / audit / close / memory / route / drift`，README 也明确了这些能力 [README.md](/Users/cc/ai/harness-before/README.md:17)。核心实现集中在 `abh/core.py`，状态机、审计、关闭、路由和漂移分析都在这里 [core.py](/Users/cc/ai/harness-before/abh/core.py:41)。

项目最大的优势是方法论清晰、dogfooding 足、闭环已能跑通。最大的风险是“文档权威性”和“工具权威性”开始分裂：`task-board` 还停在 Sprint 5，但计划和代码已经到 Sprint 7，这个问题其实已经被 memory 明确记录 [mem-doc-sync-001.md](/Users/cc/ai/harness-before/docs/memory/mem-doc-sync-001.md:18)。

**长期定位**

建议把 ABH 的长期目标定为：

> 成为 AI 协作开发中的本地优先、证据ß优先、可审计的轨迹控制层。

它不应该变成完整项目管理平台，也不该太早做 Web UI。短中期最有价值的是把“计划、验证、审计、记忆、漂移”这套工作流变成任何仓库都能无痛接入的 CLI 和 Agent 协议。

**阶段规划**

**阶段 1：恢复权威基线，稳住内核，1 个月**

目标是让项目自身不再漂移。

优先做：

1. 更新 `docs/development-roadmap.md` 和 `docs/task-board.md`，把 Sprint 6/7 和零门槛安装纳入正式路线。
2. 建立 `abh doctor` 或 `abh sync check`，检查 `.abh/*.json` 与 `docs/*.md` 是否一致。
3. 清理或关闭当前遗留的 `plan-200-demo` draft 计划，避免状态噪音。
4. 给数据对象加 schema/version 字段，为后续迁移做准备。
5. 建立 CI：运行 `unittest`，并验证 `python3 -m abh --help`、`abh plan list`。
6. 把版本号从当前 `0.1.0` 进入可解释的发布节奏，避免 README 功能与包版本脱节。

**阶段 2：Agent Protocol 基础，1-2 个月**

当前 `abh` 已经具备可参数化 CLI，但还没有 Agent Protocol：没有 MCP Server、没有统一 JSON 输出模式、没有稳定的工具 schema，所有输出仍主要面向人类阅读。下一阶段应该先把 CLI 变成 AI Agent 可程序化调用的稳定接口，再在其上封装 MCP。

建议版本：`v0.2.0`

核心能力：

1. 为核心读操作增加机器可解析输出，例如 `--json` 或等价 JSON mode。
2. 定义稳定的 Agent tool schema：plan、verify、audit、memory、route、drift、doctor 的输入、输出和错误格式。
3. 统一 CLI 返回码和结构化错误，保证 Agent 能区分成功、校验失败、业务阻断和系统错误。
4. 增加只读 MCP Server 第一版，让 Claude、Cursor 等 Agent 能读取计划、审计、记忆、漂移和当前路线。
5. 在只读 MCP 稳定后开放受控 MCP 写工具，覆盖创建 plan、状态流转、记录 verify/audit、关闭计划、写 memory 和写 drift report。
6. MCP 写工具必须显式要求 `confirm=true`，并继续复用现有 ABH 门禁，不能绕过状态机、验证、审计或 doctor。

这个阶段的目标不是替代 CLI，而是把 CLI 变成 Agent Protocol 的底座。CLI 先稳定，MCP 才有可靠封装对象。

当前状态：已完成。`plan-012-agent-protocol-foundation`、`plan-013-json-output-and-errors`、`plan-014-readonly-mcp-server` 和 `plan-015-controlled-mcp-write-tools` 已关闭，ABH 已具备 Agent Protocol 基线、JSON CLI contract、结构化错误、只读 MCP Server 和受控 MCP 写工具，可作为 `v0.2.0` 里程碑。

**阶段 3：从“记录验证”升级到“执行验证证据”，1-3 个月**

当前状态：已完成。`plan-016-verify-runner`、`plan-017-plan-update`、`plan-018-core-module-split`、`plan-019-verification-environment-metadata`、`plan-020-stage-3-functional-plan`、`plan-021-verification-trust-and-stale-detection`、`plan-022-verification-failure-classification`、`plan-023-atomic-abh-writes`、`plan-024-memory-drift-routing-module-split` 和 `plan-025-stage-3-finalization` 构成 v0.3 Verify Runner 里程碑。ABH 已具备 `abh verify run <plan>` MVP，可以执行 plan validation checklist、记录机器证据、失败阻断 ready/running plan，并支持 JSON 输出；同时已具备 `abh plan update <plan>`，可以通过 CLI 双写追加计划内容，并精确移除错误 validation checklist 条目；并已将 plan、audit、verification、memory、drift、routing 和 shared errors 逻辑拆分到领域模块，降低核心模块继续膨胀的维护风险。当前 verification 记录已经能保存 cwd、git commit、dirty status、git status hash、ABH/Python 版本、timeout、命令 argv 和 allowlisted 环境变量，并通过 trust_level、`failure_classifications` 与 `plan status --json` stale summary 提示验证证据可信等级、失败类型和过期风险；`.abh` JSON 与 Markdown 保存路径已补充原子写和本地文件锁。阶段 4 将进入 Attractor Registry。

阶段 3 的目标不是证明功能绝对正确，而是证明“在某个时间、某个 git 状态、某个本地环境下，ABH 确实执行过这些验证命令，并留下可复现、可审计的机器证据”。

建议版本：`v0.3`

核心能力：

1. `abh verify run <plan>`：按 plan 的 validation checklist 执行命令。
2. 支持保存 stdout/stderr 摘要、退出码、耗时、artifact，以及 git commit、dirty status、cwd、ABH 版本、Python 版本、命令 argv、timeout 和环境变量 allowlist。
3. 为验证结果标注可信等级，例如人工记录、本地执行、隔离目录执行和 CI 执行，避免把本地执行结果误读为绝对真实。
4. 失败时生成可审计证据；是否自动把 plan 转入 `blocked` 需要区分真实验证失败、环境失败、网络失败和 flaky failure。
5. 当 validation checklist、plan 内容或 git 状态变化后，旧 verification 应能被识别为可能 stale。
6. 支持 `abh plan update`，补充 goals/non-goals/evidence，而不是只能 create/transition。
7. 把 `core.py` 拆成 `plans.py / audits.py / memory.py / drift.py / routing.py`，否则 700 多行核心文件会越来越难控。

剩余计划队列：

1. `plan-020-stage-3-functional-plan`：完成阶段 3 剩余功能规划，不改行为。
2. `plan-021-verification-trust-and-stale-detection`：补可信等级和 stale 检测，已完成。
3. `plan-022-verification-failure-classification`：区分验证失败、环境失败、超时、递归防护等失败类别，已完成。
4. `plan-023-atomic-abh-writes`：为 `.abh` JSON 和 Markdown 双写路径补原子写或文件锁保护，已完成。
5. `plan-024-memory-drift-routing-module-split`：继续拆分 memory、drift 和 routing 模块边界，已完成。
6. `plan-025-stage-3-finalization`：收尾阶段 3，确认 v0.3 里程碑与阶段 4 启动条件，已完成。

明确不做：

1. 不把 `verify run` 声称为防篡改证明。
2. 不用验证执行器替代独立审计。
3. 不保证测试覆盖充分；审计仍需判断验证命令是否真正覆盖 exit criteria。

**阶段 4：补齐 Attractor Registry，3-4 个月**

PRD 里吸引子是第一性对象，但当前 CLI 还没有 `abh attractor` 命令，这是产品叙事和实现之间最大的缺口。

建议版本：`v0.4`

核心能力：

1. `abh attractor list/show/create/supersede`。
2. active attractor 校验：plan 进入 ready 前必须引用 active attractor。
3. 吸引子版本差异记录：为什么升级、影响范围、迁移策略。
4. route/drift 优先读取 active attractor，而不是只靠固定路径和关键词。

**阶段 5：真正独立审计，4-6 个月**

当前审计“原则上独立”，但工具无法强制；memory 已经记录过同一 session 审计的问题 [mem-independent-audit.md](/Users/cc/ai/harness-before/docs/memory/mem-independent-audit.md:18)。

建议版本：`v0.5`

核心能力：

1. `abh audit prompt <plan>`：生成独立审计 prompt。
2. `abh audit bundle <plan>`：打包审计所需证据清单。
3. `abh audit record` 增加 auditor context/source 字段。
4. 关闭计划时检查：是否有 fresh/independent 标记的通过审计。
5. 固化审计模板，避免每次靠人工记忆。

**阶段 6：漂移与记忆质量提升，6-9 个月**

现在 drift 是关键词规则，足够 MVP，但长期会误报/漏报。

建议版本：`v0.6`

核心能力：

1. drift finding 增加 severity、matched span、source excerpt、confidence。
2. memory 支持标签、状态废弃、related plan/audit 反向索引。
3. route 从关键词匹配升级为“对象图 + 简单权重排序”。
4. 增加 `abh report`：展示计划关闭率、审计驳回率、重复漂移、memory 命中情况。

**阶段 7：团队可用与生态集成，9-12 个月**

建议版本：`v1.0`

核心能力：

1. `abh init`：一键初始化任意仓库。
2. GitHub Actions 模板：PR 中自动跑 `abh doctor`、测试、漂移检查。
3. Git hooks 可选集成：关闭 plan 前检查审计和证据链。
4. 多仓库共享 attractor/memory 的导入导出。
5. 发布到 PyPI，同时保留 `uvx --from git+...` 路径。

**最重要的取舍**

不要急着做 Web UI。这个项目目前的核心价值在“仓库即真相”和“AI agent 可读写”，Web UI 过早会把精力拉到展示层。

不要把 drift 过早做成复杂智能分析。先把证据链、schema、对象关系和关闭门禁做硬，智能分析以后再增强。

不要继续让文档同步靠纪律。应该把 `docs` 与 `.abh` 一致性变成命令和 CI 门禁。
